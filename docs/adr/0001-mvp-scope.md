# ADR 0001: Границы MVP TG Order Radar

Дата: 2026-07-13

Статус: принято

## Контекст

TG Order Radar проектируется как сервис для поиска заказов на разработку сайтов в публичных Telegram-каналах и группах. Полная спецификация описывает production-ready систему, но первая рабочая версия должна быть меньше по объему и проверять главный пользовательский сценарий: найти свежий публичный заказ, отсеять нерелевантное и дубли, показать его через API и отправить уведомление в Telegram-бот.

MVP ограничивается публичными источниками Telegram. Приватные ссылки, закрытые чаты, обход CAPTCHA, антибот-защит, FloodWait и иных ограничений Telegram не входят в архитектуру и не должны поддерживаться.

## Решение

В первую версию включается только функциональность, необходимая для рабочего MVP:

- ручное добавление публичных Telegram-источников;
- проверка публичности источника;
- polling/backfill сообщений не глубже 7 дней плюс небольшой буфер;
- нормализация текста без потери исходного сообщения;
- keyword matcher и negative keyword matcher;
- rules-классификация без ML, LLM и внешних моделей;
- извлечение типа проекта, бюджета, срока и контактов;
- расчет relevance score;
- базовая дедупликация по `source_id` + `tg_message_id` и `content_hash`;
- REST API для управления источниками, просмотра заказов, статусов, избранного, словарей и экспорта в пределах MVP;
- Telegram-бот уведомлений через Bot API;
- Docker Compose для локального запуска;
- unit, integration, API и e2e-тесты в объеме, необходимом для проверки MVP-сценария.

Из первой версии явно исключаются:

- Next.js-админка;
- ML-классификация;
- LLM-этап;
- pgvector и semantic deduplication;
- импорт источников из внешних каталогов;
- мультиаккаунт и ротация Telegram-аккаунтов;
- Traefik;
- Grafana и полноценный monitoring stack;
- production-деплой.

MVP должен оставаться production-oriented по базовым инженерным решениям: миграции, идемпотентность, настройки через environment variables, структурные логи и защита секретов закладываются с начала, даже если часть production-инфраструктуры откладывается.

## Решения, которые нельзя откладывать

Идемпотентность обработки сообщений и задач должна быть заложена до реализации pipeline. Повторный сбор, повторная доставка Celery-задачи или рестарт worker не должны создавать дубли сообщений, заказов и уведомлений.

Alembic используется с момента появления схемы БД. Все изменения структуры PostgreSQL должны проходить через миграции, чтобы проект можно было поднимать с нуля и безопасно развивать.

Идентичность Telegram-источника строится вокруг стабильного `tg_peer_id`. `username` хранится и нормализуется, но не считается главным идентификатором, потому что публичный канал или группа может быть переименована.

Telegram session-файлы и session-секреты должны быть исключены из Git и храниться отдельно от кода. Реальные секреты не попадают в репозиторий; `.env.example` содержит только безопасные примеры.

`correlation_id` и структурное логирование нужны с первого этапа прикладной инфраструктуры. Без них будет сложно расследовать сбои в цепочке API, collector, Celery worker и bot.

## Последствия

Проект реализуется поэтапно: сначала фиксируются границы MVP и инфраструктурные решения, затем добавляются каркас, БД, API, collector, pipeline, дедупликация и бот. Это снижает риск попытки сразу построить весь production-проект и позволяет проверять готовность каждого этапа отдельными тестами и Docker Compose-командами.

Отложенные функции не считаются техническим долгом MVP, если их отсутствие явно отражено в документации и не ломает основной сценарий поиска, дедупликации и уведомления о свежем публичном заказе.
